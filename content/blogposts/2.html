Building a responsive analytics platform - the beginning

By: Jesse Dahl on Jan 21, 2016 responsive design angularjs

As it goes with many user-facing projects built by large corporations, proper attention and care is not given to the UX design of an application before it starts getting implemented.

So it went with Experian's AudienceView. This project’s front-end tech stack consisted of the following:

  • AngularJS
  • Bootstrap
  • LESS
  • Grunt

In this post I will discuss how I used a mobile-first process to make this application responsive.

Beginnings

The project's mockups were an incoherent, incomplete mess that no one was really responsible for, nor working on. We literally made it up as we went and built a lot of good functionality quickly.

Halfway into the project we finally managed to land a Graphic Designer! Most of the code and markup was already written! Yay!

The Mockups

Soon she (Melanie Green) produced annotated mockups (similar to the image below).

home page mockup

My task was to create a working, responsive and mobile-ready web application out of it, salvaging the existing code base and UI to do so.

The main techniques I used to make the site responsive were Media Queries and Bootstrap 3’s Grid System.

I would normally include responsive images, but this was an analytics app with 1 image on the main page, so it wasn’t very important. We had captive users who visited the app over and over.

Media Queries

Media queries are an important tool in mobile-first driven development. They allow you to specify styles based on the viewport’s dimensions. Since the project used Bootstrap 3, I decided to borrow the same breakpoints (thanks to Nicholas Cerminara for a great reference that I’ve come back to many times). Below are the breakpoints I used.

I used the mobile-first design method (why mobile-first?), which required me to make many liberties with the desktop mockups. I first divided the mockup into its main sections, then decided how each of the elements within would fall in a small phone-sized viewport.

home page sections

Mobile View

As you can see in the mobile view below, I collapsed the menu items into a “hamburger menu”.

phone view

Normally I would try to avoid hamburger menus and keep the menu items visible, but the size of the product logo and name was too large to include both, so I stuffed them into the hamburger.

The next thing you see is the dropdown that lets you run reports, the primary functionality of our application. The dropdown was just barely wide enough to display its content adequately, even after shrinking the button. Was a dropdown the right choice for this UI element? No. I think a typeahead type dropdown would probably have been easier to use, and it was certainly on my radar for the next iteration.

The next 4 buttons contained the rest of the functionality of the application, which was creating ad-hoc reports based on web traffic data married with population demographics data. If you look back at the desktop mockup, you’ll see it has the buttons side-by-side, which I stacked (a common technique for adapting content to mobile devices). Despite the stacking, all the buttons remained visible.

The only thing you can’t see above the fold is the image of our app inside of a monitor, which has limited value for repeat users.

Large Phone + Small Tablet View

first breakpoint

As you can see here, I kept the main grid layout stacked, and used the extra width to make some content wider (the “Run Report” dropdown), and unstack some other content (crosstab, digital behavior, snapshots buttons, and the help section).

You might notice that the buttons are wider in the phone version than they are at this first breakpoint. I often use a combination of min-width and max-width in px or em’s with width in percent to let an element adapt to its available width while ensuring the content within doesn’t scale down to an unusable width.

I also brought out the menu items and removed the hamburger. When possible, it is a good idea to expose available content and functionality to the user without requiring effort on the user’s part (aka interaction cost).

Large Tablet + Small Desktop View

larger mobile devices

You can see here that I finally unstacked the second main grid div. As a result, the content area for the first breakpoint was reduced. You will usually get increased vertical height along with this larger viewport, so the main functionality is still available and visible above the fold even though the buttons are again stacked.

Conclusion

Although the UX design process didn’t employ a mobile-first method, I still felt like it was beneficial to start by approaching the implementation in a mobile-first method. I did not find it difficult to create a small viewport version of the app and scale up to larger screens from there.

While it depends on the context, I would definitely lean towards using this method for creating anything that is public facing and informational, since usage is trending more and more towards mobile devices. For complicated business type web apps that require a lot of user interaction, how you decide to implement it should depend to a great deal on who your users are and how they will be accessing your webapp. More and more though, we are seeing people replacing their laptop usage with tablet usage, so making considerations for touch-based interfaces is going to remain very important.
